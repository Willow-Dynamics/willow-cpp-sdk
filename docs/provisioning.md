# Provisioning & DRM

## The Secure Default Architecture (RAM-Only)
The `.int8` files generated by the Willow Cloud Oracle contain proprietary biomechanical signatures. To prevent these from being stolen from end-user hardware, the SDK establishes a **Secure Default Architecture** utilizing `Model::load_from_memory()`.

This architecture ensures that the proprietary model binary never touches the physical hard drive. It exists only in volatile RAM and is purged immediately when the application terminates.

### Implementation (The Client Module)
The included `Client` module uses `libcurl` to securely fetch the encrypted model directly into an ephemeral RAM buffer. 

* **Prerequisite:** Compile your project with `-DWILLOW_ENABLE_CURL` and link `-lcurl`.

```cpp
#include <cstdlib>
#include <iostream>
#include "willow.hpp"

// Example: Securely fetching and loading a model
void provision_model() {
    try {
        // 1. Configure the Client (Best Practice: Use Environment Variables)
        // Never hardcode API keys in production source code.
        const char* key = std::getenv("WILLOW_API_KEY");
        const char* cid = std::getenv("WILLOW_CUSTOMER_ID");
        
        if (!key || !cid) {
            throw std::runtime_error("Missing WILLOW_API_KEY or WILLOW_CUSTOMER_ID in environment.");
        }
        
        willow::Client client(key, cid);

        // 2. Fetch the model into a std::vector<uint8_t>
        // This performs an HTTPS GET request to the Willow Cloud Oracle.
        // The data resides ONLY in this stack-allocated buffer.
        std::vector<uint8_t> secure_buffer = client.fetch_model("tactical_reload");

        // 3. Provision the Model from RAM
        // The SDK parses the header and de-quantizes the signature in memory.
        willow::Model model = willow::Model::load_from_memory(secure_buffer);
        
        std::cout << "Model loaded successfully. Version: " << model.version << "\n";

    } catch (const std::exception& e) {
        std::cerr << "Provisioning failed: " << e.what() << "\n";
    }
}
```

## The .int8 Binary Header
If you are auditing the code or building a custom parser, understand that the `.int8` file contains a **24-byte strict little-endian header**:

1. **Bytes [0-3]:** `uint32` - Magic Version (40)
2. **Bytes [4-7]:** `uint32` - Zone Bitmask
3. **Bytes [8-11]:** `float32` - Quantization Scale Factor
4. **Bytes [12-15]:** `float32` - Overlap Tolerance
5. **Bytes [16-19]:** `float32` - DTW Sensitivity
6. **Bytes [20-23]:** `float32` - Tempo Variance

The remainder of the file (Bytes 24+) is the Int8 quantized matrix. The SDK automatically de-quantizes this back to Float32 using the `scale_factor` during the `load_from_memory` step.

## Local File Loading
Because this SDK is open-source under the MIT License, we provide `Model::load_from_file()` for specific use cases:
1. **Local Prototyping:** Testing your integration without hitting network API limits.
2. **Secure Internal Servers:** Deploying on backend infrastructure where you control physical access to the drive.

**Security Warning:** Do not use `load_from_file` in distributed consumer products (e.g., IoT devices, Robots shipped to customers). Doing so exposes your proprietary `.int8` models to trivial theft.
